\title{Custom eFPGA Layout Synthesis for Logic Masking via Reinforcement Learning}

\begin{document}

\begin{abstract}
Embedding a small reconfigurable fabric into an ASIC to implement one
security-critical IP block is a known defense against reverse engineering,
since the protected function is absent from the fixed silicon until
configured. Because this use case targets one known netlist rather than
general-purpose reconfigurability, a custom-tailored fabric layout can
recover much of the area/delay/power overhead a generic FPGA would
otherwise impose. Prior work (GOLDS and a follow-up NSGA-II-based method)
searches for such layouts with an evolutionary algorithm, re-initialized
from scratch for every new protected netlist. We instead train a
reinforcement-learning policy, conditioned on a graph-neural-network
encoding of the target netlist, to choose DSP/BRAM placement and fabric
aspect ratio, evaluated directly against the real VTR synthesis-to-routing
flow. On a controlled, apples-to-apples comparison against our own
from-scratch genetic algorithm under an identical oracle and metric, the RL
policy reaches 74.06\% ADP reduction versus the GA's 68.17\%. Jointly
trained across 11 real circuits, the policy reaches a 35.7\% aggregate
in-pool ADP reduction. Recomputed on GOLDS' own published metric and
baseline architecture, a genuine zero-shot evaluation (no training on the
target benchmark at all) reaches 37.4\% versus GOLDS' published
$\sim$35\%. Using three independent training seeds, we additionally
characterize the reliability of these results: in-pool performance is
considerably more seed-sensitive than zero-shot generalization is, with two
specific in-pool benchmarks varying by over 25 points across seeds while
held-out benchmarks vary by no more than about 5 points each. We report
this honestly as a finding about the current approach's limits, not smooth
it into a single point estimate.
\end{abstract}

\section{Introduction}
\label{sec:intro}

Embedding a small, reconfigurable fabric into an ASIC to implement one
security-critical IP block â€” so that the function is not recoverable from the
silicon layout or GDSII without the bitstream â€” is a known defense against
reverse engineering and IP theft~\cite{golds}. A generic, general-purpose
FPGA architecture is built to serve arbitrary circuits, and is therefore
over-provisioned â€” extra LUTs, routing, and DSP/BRAM capacity â€” for any one
fixed design that will only ever run a single bitstream. Because the eFPGA
masking use case targets exactly one known application on a small embedded
fabric, rather than general-purpose reconfigurability, a custom-tailored
architecture and placement, sized and arranged specifically for that one
netlist, is viable, and can recover much of the area/delay/power overhead a
generic FPGA would otherwise impose. We use the product of area, delay, and
power (ADP) as our optimization target throughout this paper because it is
precisely the size of this overhead â€” the ``masking tax'' of moving a
protected block onto reconfigurable fabric instead of hardening it directly
into standard cells.

Two prior papers establish that per-instance search over DSP/BRAM placement
(and, in one case, CLB/arithmetic-block sizing) can substantially reduce this
tax: GOLDS~\cite{golds} uses a genetic algorithm with an Area$\times$Delay
fitness function and reports up to $\sim$35\% ADP improvement on the
\texttt{diffeq1} benchmark; a follow-up applies NSGA-II to a larger,
multi-objective search space that additionally co-evolves CLB LUT size and
custom arithmetic-block sizing~\cite{nsgaii}. Both, however, are \emph{testbench-specific}: the genetic algorithm's
population is re-initialized and re-evolved from scratch for every new
protected IP block, and every generation of every run requires its own full,
expensive VTR evaluation. Nothing learned while optimizing one netlist's
layout carries over to the next netlist â€” each new target benchmark pays the
full search cost again, with no way to amortize the work already done on
previous designs.

This paper asks whether a \emph{learned policy} can replace that
per-instance search loop, rather than just compete with it once per
benchmark. RL placing blocks onto an \emph{already-fixed} FPGA fabric is not
new; what we ask is whether the same RL machinery can instead be used to
\emph{construct} the custom fabric layout itself â€” choosing DSP/BRAM
placement and aspect ratio for a small, embedded FPGA built around one
specific netlist, exactly the regime in which a per-instance, testbench-fixed
GA has so far been the only approach. Concretely, we pose two research
questions:
\begin{itemize}
  \item \textbf{RQ1.} Reinforcement learning has been used to lay out blocks
        on a \emph{fixed} FPGA fabric; can the same RL formulation instead be
        used to determine the fabric's custom layout itself, for a given
        netlist â€” the embedded-FPGA logic-masking setting where a
        purpose-built, netlist-specific architecture is viable in the first
        place?
  \item \textbf{RQ2.} Can such an RL agent be \emph{benchmark-agnostic} â€”
        pretrained across a diverse pool of netlists so that the same
        checkpoint speeds up layout construction for a genuinely new,
        previously unseen custom benchmark, either directly (zero-shot) or
        after a short fine-tuning pass, rather than requiring a fresh search
        from scratch for every new protected IP block the way GOLDS and the
        NSGA-II follow-up do?
\end{itemize}
We train a single reinforcement-learning policy, conditioned on a
graph-neural-network encoding of the target netlist, to place DSP/BRAM hard
blocks and choose a fabric aspect ratio, evaluated directly against the real
VTR (Yosys + VPR) synthesis-to-routing flow rather than a learned or analytic
proxy. Because the same checkpoint can be applied to a netlist it has never
seen, without re-running any search, this amortizes the cost of finding a
good layout across deployments â€” the practical advantage we are after is not
necessarily winning every single-instance ADP comparison outright, but being
fast and strong on a genuinely new protected IP block.

We report three results. First, on a controlled, fully apples-to-apples
comparison â€” our own genetic algorithm and our RL policy evaluated under the
\emph{identical} oracle, ADP metric, and traditional baseline, on a benchmark
held out of RL training â€” RL reaches 74.06\% ADP reduction versus the GA's
68.17\% (Section~\ref{sec:results-ga}). Second, jointly trained across 11
benchmarks, the policy achieves a 27.35\% aggregate in-pool ADP reduction
averaged over three independent training seeds, with one persistent laggard
benchmark and two specific benchmarks responsible for almost all of the
seed-to-seed spread (Section~\ref{sec:results-inpool}). Third, recomputed on
GOLDS' own published 2-term Area$\times$Delay metric and baseline
architecture, a zero-shot evaluation â€” a policy trained only on
\texttt{diffeq2}, applied directly to \texttt{diffeq1} with no
benchmark-specific training at all â€” reaches 37.4\% versus GOLDS' published
$\sim$35\% (Section~\ref{sec:results-golds}). We also report a less
convenient finding alongside these: across the same three seeds, zero-shot
generalization to six held-out benchmarks is consistently positive on every
benchmark, with far more uniform seed-to-seed spread (standard deviation
2.0--5.5 points per benchmark) than in-pool fitting shows (0.1--12.0,
dominated by two specific benchmarks) (Section~\ref{sec:results-zeroshot});
we present this honestly as a characterization of the current approach's
reliability rather than smoothing it into a single point estimate.

This paper sits at the intersection of two lines of work that have not, to
our knowledge, been connected before (Section~\ref{sec:related}): custom
eFPGA layout search for logic masking, which to date has used only
per-instance evolutionary search~\cite{golds,nsgaii}, and learned policies for
physical design, which to date have targeted ASIC macro placement at a much
larger scale and through a learned reward proxy rather than the true
toolchain~\cite{alphachip,chipformer}. We bring the amortization argument
from the latter line of work to the former problem, evaluated end-to-end
against the real VTR oracle the former line of work already established as
the right fidelity bar for this problem.

Concretely, this paper contributes:
\begin{itemize}
  \item A reinforcement-learning formulation of custom eFPGA DSP/BRAM
        placement and aspect-ratio selection, with a GCN-based netlist
        encoding (Section~\ref{sec:gnn}) and a fixed-size policy that remains
        loadable against benchmarks never seen during training
        (Section~\ref{sec:universe}), evaluated directly against the same
        VTR-based oracle used by prior eFPGA-layout search work.
  \item A fully controlled, apples-to-apples comparison against a
        from-scratch genetic algorithm under an identical evaluation oracle,
        ADP metric, and traditional baseline, on a benchmark held out of RL
        training: 74.06\% ADP reduction for RL versus 68.17\% for the GA
        (Section~\ref{sec:results-ga}).
  \item A joint 11-benchmark training result (27.35\% aggregate in-pool ADP
        reduction, averaged over three independent seeds) (Section~\ref{sec:results-inpool}).
  \item A literature-facing comparison against GOLDS' own published metric
        and baseline architecture, recomputed precisely on a genuine
        zero-shot evaluation with no benchmark-specific training at all:
        37.4\% versus GOLDS' published $\sim$35\% (Section~\ref{sec:results-golds}).
  \item A three-seed characterization of result reliability across both
        in-pool and zero-shot settings, surfacing a clear asymmetry â€” in-pool
        fitting is far more seed-sensitive (driven by two specific
        benchmarks varying by $\sim$12 points across seeds) than zero-shot
        generalization is (uniformly 2--5.5 points across all six held-out
        benchmarks) â€” reported as a finding about the current approach's
        limits, not smoothed into a single optimistic point estimate
        (Section~\ref{sec:results-zeroshot}, Section~\ref{sec:results-tradeoff}).
\end{itemize}

\section{Background and Related Work}
\label{sec:related}

\subsection{eFPGA-Based Logic Masking}

Embedding a reconfigurable fabric into an ASIC to hide one IP block's
function behind a bitstream is a hardware-security technique distinct from
standard logic locking or camouflaging, since the protected function is not
merely obfuscated in the netlist but physically absent from the fixed
silicon until configured. GOLDS~\cite{golds} introduced custom-tailored
eFPGA layout design for this purpose: a genetic algorithm searches over
DSP/BRAM tile placement on a fixed-size fabric, with an
Area$\times$Delay fitness function evaluated through the real VTR flow, and
reports up to $\sim$35\% ADP-relevant improvement over a traditional
island-style layout on \texttt{diffeq1}. A follow-up~\cite{nsgaii} extends
this to a multi-objective NSGA-II search that additionally co-evolves CLB
LUT size and custom DSP/BRAM block sizing â€” not just their placement â€”
optimizing delay and power jointly and reporting up to $\sim$44.6\%
improvement on a larger design-space. Both papers' searches are run fresh,
from a random population, for every new protected design; neither carries
information learned on one IP block forward to the next. This paper targets
the same problem GOLDS defines (custom DSP/BRAM placement plus aspect ratio,
evaluated through the same VTR-based oracle) and treats it as the
narrower, more directly comparable target for our controlled experiments
(Section~\ref{sec:results-ga}), while citing the NSGA-II extension as
context for the larger design space a future version of this approach could
also cover (Section~\ref{sec:conclusion}).

\subsection{Learned Policies for Physical Design}

Reinforcement learning has been applied to chip-scale physical design
problems structurally similar to ours. AlphaChip~\cite{alphachip} trains an
RL agent to place macro blocks for ASIC floorplanning, using a learned
reward proxy in place of a full place-and-route evaluation per step, and
reports that the resulting policy transfers across chip designs without
retraining from scratch â€” the same amortization argument we make for eFPGA
placement, applied to a different physical-design problem and at a much
larger design scale (full TPU floorplans rather than DSP/BRAM tile
placement on a small embedded fabric). ChiPFormer~\cite{chipformer} instead
poses macro placement as an offline RL problem over a transformer policy,
trained on a fixed dataset of prior placements rather than online
environment interaction. Both works motivate the same underlying claim this
paper tests in a different, smaller-scale setting: that a trained policy can
out-amortize per-instance search once a population of designs needs to be
placed, rather than re-running search from scratch on each one. Unlike
AlphaChip, we evaluate every candidate placement against the true VTR
synthesis-to-routing flow rather than a learned proxy, trading evaluation
speed for evaluation fidelity â€” appropriate at our problem's much smaller
scale, where a full VTR run completes in seconds to low minutes
(Section~\ref{sec:oracle}).

\subsection{VTR/VPR Background}

We build directly on the open-source Verilog-to-Routing
(VTR)~\cite{vtr,vpr} toolchain: Yosys performs technology-independent
synthesis, and VPR performs packing, placement, and routing against an
architecture described in an XML schema specifying tile types, their
positions on the fabric, and their connectivity. This architecture XML is
the same artifact GOLDS and the NSGA-II follow-up generate by hand-coded
search and that we generate by policy rollout (Section~\ref{sec:oracle}); VTR's
willingness to accept an essentially arbitrary heterogeneous tile layout,
rather than only the regular island-style grids it ships by default, is what
makes the entire custom-eFPGA-layout line of work â€” GOLDS, the NSGA-II
follow-up, and this paper â€” possible in the first place.

\section{Methodology}

\subsection{Problem Formulation}
\label{sec:problem}

Given a fixed eFPGA fabric template parameterized by a tile grid of width $W$ and
height $H$, and a netlist requiring $n_{\mathrm{dsp}}$ DSP and $n_{\mathrm{bram}}$
BRAM hard blocks, our goal is to choose a placement
$\pi = (\mathrm{ar}, \{(x_i,y_i)\}_{i=1}^{n_{\mathrm{dsp}}+n_{\mathrm{bram}}})$ â€”
an aspect ratio $\mathrm{ar} \in \mathcal{R}$ for the fabric plus a tile coordinate
for every hard block â€” that minimizes the post-synthesis, post-place-and-route
cost of implementing that netlist on the resulting custom architecture. We use
the product of routing area, critical-path delay, and power (ADP) as the cost,
since this directly reflects the silicon and timing overhead of moving the
protected IP block onto reconfigurable fabric:
\begin{equation}
\label{eq:adp}
  \mathrm{ADP}(\pi) = \mathrm{Area}(\pi) \times \mathrm{Delay}(\pi) \times \mathrm{Power}(\pi),
\end{equation}
where each term is obtained by running the candidate placement through the full
VTR (Yosys + VPR) synthesis-to-routing flow (Section~\ref{sec:setup}). We report
results as percentage ADP reduction relative to a traditional, non-customized
island-style eFPGA baseline at the same logic capacity.

\subsection{RL Formulation}
\label{sec:rl}

We pose placement as a finite-horizon sequential decision process and train a
policy with Proximal Policy Optimization (PPO), using the maskable variant
(\texttt{sb3\_contrib}'s \texttt{MaskablePPO}) so that the action space can be
restricted to currently-legal tile coordinates at every step.

\textbf{Episode structure.} An episode places exactly one benchmark's blocks
onto the shared canvas. Step 0 selects the fabric aspect ratio from a fixed
discrete set $\mathcal{R}$; subsequent steps place each DSP, then each BRAM
instance, one per step, in descending order of shared-net connectivity. Once
every block is placed, the environment bakes the resulting layout into an
architecture XML and constraints file (via Jinja2 templating) and invokes the
VTR flow to obtain $\mathrm{Area}(\pi)$, $\mathrm{Delay}(\pi)$, and
$\mathrm{Power}(\pi)$; all intermediate steps receive zero reward.

\textbf{Action space.} The action space is $\mathrm{Discrete}(W_{\max} \times
H_{\max})$, fixed across all benchmarks in a training run (Section~\ref{sec:universe}).
At step 0 an action index selects an entry in $\mathcal{R}$; at later steps an
action index decodes to a tile coordinate $(x,y)$ via a fixed stride on
$H_{\max}$, independent of which benchmark is active. Action masking â€” keyed
off the \emph{active} benchmark's real width/height and the blocks already
placed â€” restricts the policy to legal, non-overlapping, in-bounds coordinates
for that specific benchmark and step.

\textbf{Reward.} The terminal reward is a weighted sum of negative log-ratios
against the traditional baseline for the active benchmark:
\begin{equation}
\label{eq:reward}
  r = -\Big(
    w_a \log\tfrac{\mathrm{Area}}{\mathrm{Area}_0} +
    w_d \log\tfrac{\mathrm{Delay}}{\mathrm{Delay}_0} +
    w_p \log\tfrac{\mathrm{Power}}{\mathrm{Power}_0} +
    w_w \log\tfrac{\mathrm{WL}}{\mathrm{WL}_0}
  \Big),
\end{equation}
where $\mathrm{Area}_0,\mathrm{Delay}_0,\mathrm{Power}_0,\mathrm{WL}_0$ are the
traditional-baseline metrics for that benchmark and $w_a,w_d,w_p,w_w$ are
configurable weights (Section~\ref{sec:training}). When $w_a=w_d=w_p=1$ and
$w_w=0$, $r$ reduces to $\log(\mathrm{ADP}_0/\mathrm{ADP})$ â€” i.e.\ the reward
\emph{is} the log of the ADP-improvement ratio in Eq.~\ref{eq:adp}, making
reward and our headline metric directly comparable. An invalid choice
(degenerate aspect ratio, out-of-bounds or overlapping placement, or a VTR
flow failure) terminates the episode early with a fixed penalty reward of $-10$.

\subsection{GCN Feature Extractor}
\label{sec:gnn}

To let a single policy generalize across benchmarks with different netlist
sizes, hard-block counts, and grid dimensions, we first reduce each
benchmark's full netlist graph to a compact, fixed-schema representation: one
node per DSP/BRAM hard-block instance, plus one aggregate ``fabric'' node
summarizing the remaining standard-cell logic. Each node carries a 4-dimensional
feature vector \texttt{[is\_dsp, is\_bram, is\_fabric, net\_count\_normalized]}.
Edges between two hard-block nodes are weighted by their raw shared-net count;
edges between a hard-block node and the fabric node are weighted by the summed
shared-net count across all standard-cell members that block touches. Because
these two edge kinds live on very different numeric scales, we normalize each
into $[0,1]$ separately with a log compression, $w_{\mathrm{norm}} =
\log(1+w)/\log(1+c)$, using a per-kind ceiling $c$ calibrated to the observed
maximum across the training benchmark pool â€” a single shared linear ceiling
either saturates the larger-magnitude fabric edges or crushes the
direct edges near zero.

This reduced graph is padded to a fixed $(\mathrm{MAX\_NODES},
\mathrm{MAX\_EDGES})$ â€” computed once over the full benchmark \emph{universe}
(Section~\ref{sec:universe}) â€” and fed to a small GNN feature extractor shared
by the policy and value heads: two \texttt{GCNConv} layers (hidden width 64)
produce a per-node embedding $h_i$; a global mean pool over $h$ gives a
whole-netlist embedding, and the embedding of the specific node being placed
this step (looked up via \texttt{current\_block\_idx}) gives a focused,
per-step embedding. These are concatenated with (i) a small CNN over the
occupancy grid (4 channels: occupancy, block-type hint, aspect ratio, and net
count, each padded with an out-of-canvas sentinel outside the active
benchmark's real footprint) and (ii) the active benchmark's normalized
$(\mathrm{width}, \mathrm{height})$, then projected to a 128-dimensional joint
feature vector consumed by the PPO policy and value heads. The whole extractor
is trained end-to-end under the standard PPO loss; there is no separate
pretraining stage for the GNN.

\subsection{Universe Superset and Action Masking}
\label{sec:universe}

A policy network's input/output dimensions are fixed at construction time, but
we want one checkpoint to remain loadable against benchmarks it was never
trained on, including ones discovered after training. We address this by
sizing every dimension-dependent quantity â€” canvas $(W_{\max}, H_{\max})$,
$\mathrm{MAX\_NODES}$, $\mathrm{MAX\_EDGES}$ â€” from a benchmark
\emph{universe} that is a deliberate superset of the benchmarks actually
sampled during training: it can include benchmarks reserved purely for
zero-shot evaluation, so long as none of them exceeds the resulting shared
dimensions. Per-episode action masking (Section~\ref{sec:rl}) and the
\texttt{valid\_wh} observation are what let the same fixed-size network
operate correctly on each benchmark's much smaller real footprint within that
shared canvas.

\subsection{Training Setup}
\label{sec:training}

Each training run samples uniformly across the 11-benchmark training pool
(Table~\ref{tab:benchmarks}) episode-by-episode, so a single policy is
updated jointly on all 11 circuits rather than one at a time. We use
\texttt{MaskablePPO} with 24 parallel environments, 48 rollout steps per
environment per update, batch size 144, 15 epochs per update, entropy
coefficient 0.01, and reward weights $w_a=w_d=w_p=1$, $w_w=0$ (Eq.~\ref{eq:reward}),
giving reward $=\log(\mathrm{ADP}_0/\mathrm{ADP})$. Each run trains for
13{,}000 episodes (capped independently of total timesteps) with a
per-episode VTR timeout of 300s to keep a degenerate early-exploration
episode from stalling the other 23 parallel workers. We report results from
three independent training seeds (7, 42, 123) under this identical
configuration, used throughout Section~\ref{sec:results} to characterize
result reliability rather than report a single run's numbers as ground
truth.

\section{Experimental Setup}
\label{sec:setup}

\subsection{Evaluation Oracle}
\label{sec:oracle}

Every candidate placement is scored by the true downstream toolchain, not a
proxy. A placement (aspect ratio + hard-block coordinates) is rendered into a
complete VTR architecture XML and a VPR placement-constraints file via Jinja2
templates parameterized by tile coordinates, block type, and fabric aspect
ratio. This architecture and the benchmark's Verilog are then run through the
full VTR flow â€” Yosys-based synthesis, VPR packing, placement, and routing,
followed by VPR's power estimator against a 45nm PTM technology file â€” and we
extract routing area, critical-path delay, dynamic power, and wirelength from
the resulting VPR reports. Evaluated (placement, benchmark) pairs are cached,
since the synthesis-to-routing flow is the dominant cost of both training and
evaluation (seconds to low minutes per call depending on benchmark size).

\subsection{Baseline}
\label{sec:baseline}

Our traditional baseline is VTR's default island-style architecture template
(\texttt{k6\_frac\_N10\_mem32K\_40nm}: 6-input LUTs, cluster size 10, 32K-bit
BRAMs, 40nm) at matched logic capacity for each benchmark â€” the same template
used by GOLDS~\cite{golds} as its traditional-layout comparison point, which
lets us compare percentage improvements directly against theirs on a shared
starting point (Section~\ref{sec:results-golds}).

\section{Results}
\label{sec:results}

\subsection{In-Pool Performance}
\label{sec:results-inpool}

Table~\ref{tab:inpool} reports per-benchmark ADP reduction for the
11-benchmark policy directly on its own training set, across three
independent training seeds (7, 42, 123; Section~\ref{sec:training}), as
mean $\pm$ population standard deviation. Each seed's own pooled aggregate
â€” one minus the ratio of summed ADP across all 11 benchmarks to summed
baseline ADP, \emph{not} a simple mean of per-benchmark percentages â€” is
35.73\%, 23.11\%, and 23.97\% respectively (mean 27.35\%, $\pm$5.40).
\texttt{raygentop} is a persistent laggard in all three seeds, tightly
clustered just below zero ($-2.01\% \pm 0.22$), which we attribute
tentatively to its 6-DSP/1-BRAM profile being a thin slice of the joint
training distribution rather than a fundamental failure mode. Two
benchmarks, \texttt{or1200} and \texttt{boundtop}, are responsible for
almost all of the seed-to-seed instability: standard deviations of 11.82
and 12.03 respectively, an order of magnitude larger than every other
benchmark in the pool (all $\le 3.3$). No single seed wins on every
volatile benchmark â€” seed 42 is strongest on both \texttt{or1200} and
\texttt{boundtop}, while seed 123 and seed 7 each lead on at least one
other benchmark by a smaller margin â€” which argues against "some seeds are
simply better policies" and toward benchmark-specific convergence luck
within a fixed episode budget (Section~\ref{sec:results-tradeoff}).

\subsection{GA Head-to-Head Comparisons}
\label{sec:results-ga}

To establish a fully controlled, apples-to-apples comparison against
evolutionary search â€” not subject to any metric or baseline-matching caveat â€”
we implemented our own genetic algorithm under the identical evaluation
oracle, ADP metric, and traditional baseline as our RL pipeline, and ran it on
\texttt{custom\_macbuf} (held out of RL training; Table~\ref{tab:benchmarks}):
roulette-wheel selection, single-point crossover, per-gene mutation over both
hard-block coordinates and fabric aspect ratio, population 5, capped at 500

reaches \textbf{74.06\%} ADP reduction versus the GA's \textbf{68.17\%} on

\subsection{Head-to-Head Against GOLDS}
\label{sec:results-golds}

As a secondary, literature-facing comparison, we recompute a genuine
zero-shot result â€” a policy trained only on
\texttt{diffeq2}, evaluated zero-shot on \texttt{diffeq1} with no training on
it whatsoever â€” on GOLDS'~\cite{golds} own 2-term Area$\times$Delay metric
(rather than our 3-term Area$\times$Delay$\times$Power metric used
elsewhere), under the same traditional baseline architecture GOLDS uses. This
yields a \textbf{37.4\%} reduction versus GOLDS' published best-of-population
result of \textbf{$\sim$35\%} on the same benchmark â€” a real, if narrower,
margin, achieved with zero benchmark-specific training. We report this
result separately from Section~\ref{sec:results-ga} precisely because it
required recomputing on a different metric than the rest of this paper;
nowhere else do we compare 2-term GA numbers against our 3-term RL numbers
directly.

\subsection{Zero-Shot Generalization}
\label{sec:results-zeroshot}

Table~\ref{tab:zeroshot} reports ADP reduction on all six held-out benchmarks
(Table~\ref{tab:benchmarks}), across the same three training seeds as
Section~\ref{sec:results-inpool}, evaluated with zero benchmark-specific
training. Two of the six (\texttt{softmax}, \texttt{reduction\_layer}) were
part of the training-time \emph{universe} used to size the policy's tensors
(Section~\ref{sec:universe}) but never sampled during training; the other
four were entirely outside the universe as well, a strictly harder
generalization test. Every benchmark finishes positive under every seed â€”
no sign flips at all, unlike the in-pool result â€” and per-benchmark standard
deviations are uniformly moderate (2.0--5.5), with no single benchmark
approaching the instability seen in \texttt{or1200}/\texttt{boundtop}
in-pool. The overall average across all six benchmarks and three seeds is
\textbf{31.23\%}. The strongest, most reliable benchmarks
(\texttt{mkDelayWorker32B}, \texttt{custom\_macbuf}) are also the most
seed-stable in relative terms (coefficient of variation 5--10\%), while the
weakest (\texttt{softmax}, \texttt{arm\_core}) are proportionally the least
certain (coefficient of variation 22--34\%) â€” the absolute spread is small
everywhere, but on a small mean that spread represents a much larger
fraction of the result.

\textbf{A caveat on what this table measures.} Every number above uses
\emph{deterministic} policy decisions (the policy's single most-likely
action at each step). As a preliminary check on one seed, we additionally
sampled 10 \emph{stochastic} rollouts per benchmark (sampling from the
policy's action distribution instead of always taking its top choice); the
average ADP reduction across those samples was negative for 4 of 6
benchmarks, even though the deterministic result for that same seed and
benchmark was positive in every case. This indicates the deterministic
numbers in this section represent the policy's best-case, most-confident
behavior, not necessarily what a stochastic deployment would see on
average. We flag this explicitly as a limitation rather than omit it:
Table~\ref{tab:zeroshot} should be read as "what the policy's top choice
achieves under each seed," not as an expectation over repeated sampling.

\subsection{Fine-Tuning Efficiency}
\label{sec:results-finetune}

Separately from cross-benchmark transfer, we characterize how efficiently a
multi-benchmark policy continues learning on a benchmark it has already
partially seen during joint training (\texttt{diffeq1}, within the
11-benchmark pool), compared to training an identical architecture from
scratch on that benchmark alone, both for 6{,}000 episodes under identical
hyperparameters and separate VTR caches. The fine-tuned run dominates typical
reward almost throughout (0.81 vs.\ 0.65 in the final 200-episode window,
still climbing for scratch) and reaches its convergence plateau by roughly
episode 1{,}600--1{,}800 ($\approx$7 minutes wall-clock), after which it
barely moves; scratch never plateaus within its full 6{,}000-episode budget.
The mechanism is policy entropy collapse: the fine-tuned run's entropy falls
from 2.00 to 0.79 over the run (vs.\ scratch's 4.63 to 2.42), so only 31.5\%
of its episodes are genuinely novel VTR evaluations versus 94.5\% for
scratch â€” it converges to a good region and stops exploring sooner, which is
a sample-efficiency win here but, as Section~\ref{sec:results-tradeoff}
discusses, the same mechanism caps how much further exploration a fine-tuned
policy is willing to do beyond its pretrained prior. We note this result
characterizes \emph{continued learning} on a partially-known benchmark, not
cross-benchmark transfer; the latter is addressed in
Section~\ref{sec:results-zeroshot}.

\subsection{Seed Variance: In-Pool vs.\ Zero-Shot}
\label{sec:results-tradeoff}

Across three independent training seeds, the 11-benchmark policy shows a
clear and somewhat counterintuitive asymmetry: \emph{in-pool} performance is
considerably more seed-sensitive than \emph{zero-shot} performance is. In
nine of the eleven training benchmarks, standard deviation across seeds is
small (0.10--3.33; Table~\ref{tab:inpool}); but two benchmarks,
\texttt{or1200} and \texttt{boundtop}, swing by roughly 12 points each â€” an
order of magnitude larger than the rest of the pool â€” and these two alone
are responsible for almost the entire spread in the pooled aggregate
(35.73\% / 23.11\% / 23.97\% across the three seeds). On the six held-out
zero-shot benchmarks, by contrast, every standard deviation falls in a much
narrower, more uniform band (2.0--5.5; Table~\ref{tab:zeroshot}), with no
single benchmark behaving like \texttt{or1200} or \texttt{boundtop} does
in-pool.

We do not yet have a confirmed mechanism for this asymmetry, but offer a
plausible explanation: in-pool performance is exposed to which specific
benchmarks a given seed's policy happens to converge on slowly within a
fixed, shared episode budget across all 11 benchmarks sampled jointly,
while zero-shot performance instead reflects a more general property of
the learned netlist representation (Section~\ref{sec:gnn}) that does not
depend on any one training benchmark's individual convergence trajectory.
Section~\ref{sec:results-finetune}'s entropy-collapse mechanism plausibly
compounds the in-pool effect specifically: a seed that converges early on
most benchmarks has fewer of its limited episodes left for a
slow-converging benchmark like \texttt{or1200} to catch up, while a seed
that explores longer everywhere may give that one benchmark more of a
chance. Critically, no single seed wins on every volatile in-pool benchmark
(seed 42 leads on both \texttt{or1200} and \texttt{boundtop}, but seed 123
and seed 7 each lead elsewhere by smaller margins) â€” this argues against
"some seeds are simply better policies" and toward per-benchmark convergence
luck specifically. This remains a characterization at $n=3$ seeds, not a
rigorously tested hypothesis; a fourth or fifth seed would meaningfully
sharpen whether the asymmetry itself is stable or partly an artifact of
still having too few seeds.

\section{Conclusion}
\label{sec:conclusion}

We asked whether a learned policy can do better than restarting evolutionary
search from scratch for every new protected IP block in custom eFPGA-based
logic masking. Under a fully controlled, apples-to-apples comparison â€” our
own genetic algorithm and our RL policy on the identical benchmark,
evaluation oracle, ADP metric, and traditional baseline â€” RL wins: 74.06\%
ADP reduction versus the GA's 68.17\% (Section~\ref{sec:results-ga}). Jointly
trained across 11 benchmarks, the same policy reaches a 27.35\% aggregate
in-pool ADP reduction averaged over three independent seeds
(Section~\ref{sec:results-inpool}). Recomputed
on GOLDS' own published metric and baseline architecture, a genuinely
zero-shot evaluation â€” no training on the target benchmark at all â€” reaches
37.4\% versus GOLDS' published $\sim$35\% (Section~\ref{sec:results-golds}).

We also reported a result that complicates a clean amortization story:
across three training seeds, in-pool fitting is considerably more
seed-sensitive than zero-shot transfer is
(Section~\ref{sec:results-tradeoff}) â€” the in-pool pooled aggregate ranges
from 23.11\% to 35.73\% across seeds, with two specific benchmarks
(\texttt{or1200}, \texttt{boundtop}) responsible for almost the entire
spread, while every one of the six zero-shot benchmarks varies by no more
than about 5.5 points across the same three seeds
(Section~\ref{sec:results-zeroshot}). We also found that this section's
deterministic (greedy) zero-shot numbers can diverge sharply from a
policy's \emph{stochastic} behavior â€” a preliminary 10-rollout check on one
seed found average ADP reduction negative for 4 of 6 benchmarks under
stochastic sampling, despite every benchmark's deterministic result being
positive. We see both of these not as weaknesses to be written around but
as the most important findings for anyone trying to reproduce or build on
this line of work â€” a single-seed or single-episode result, in either
direction, should not be taken as a stable characterization of the policy's
behavior.

Several gaps remain. Every result in this paper comes from three
seeds, not the larger seed-averaged statistics with tight error bars a
fully rigorous claim would eventually need â€” itself a direct consequence of
the variance finding above, and the most immediate item for any follow-up,
along with a systematic stochastic-rollout evaluation across all three
seeds rather than the single-seed preliminary check reported here. The action space we search
covers DSP/BRAM placement and aspect ratio only; the NSGA-II
follow-up~\cite{nsgaii} additionally co-evolves CLB LUT size and custom
arithmetic-block sizing, and extending our formulation to that larger space
is a natural next step, now that the GCN-based representation
(Section~\ref{sec:gnn}) already generalizes across netlists of different
size and resource mix. Our benchmark pool, while including hand-written
probes specifically designed to stress DSP/BRAM combinations absent from
real circuits, does not yet cover the full set GOLDS and the NSGA-II
follow-up report on (e.g.\ \texttt{sha}, the Koios deep-learning suite);
broadening coverage would let every comparison in this paper be made
directly rather than through the single-benchmark, metric-converted
comparison in Section~\ref{sec:results-golds}.
